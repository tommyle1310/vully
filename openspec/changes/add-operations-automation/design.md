### 1.2 Payment Reference Format

To link incoming payments to invoices, we generate a unique payment reference:

```
Pattern: VULLY-{entity}-{id-suffix}

Examples:
  - Invoice:   VULLY-INV-a1b2c3   (last 6 chars of UUID)
  - Contract:  VULLY-CTR-d4e5f6
```

**Webhook Description Parsing:**
```typescript
// Bank transfer description: "VULLY-INV-a1b2c3 Thanh toan tien phong T4/2026"
const regex = /VULLY-(INV|CTR)-([A-Za-z0-9]{6})/;
```

### 1.3 Idempotency Strategy with Database Transaction

| Field | Source | Purpose |
|-------|--------|---------|
| `external_transaction_id` | Gateway's tx ID | Prevent duplicate processing |
| `payment_reference` | Generated by Vully | Map webhook to invoice/contract |
| `raw_gateway_response` | Full webhook payload | Audit and dispute resolution |

**вҡ пёҸ Critical: Atomic Transaction Requirement**
Gateways may send duplicate webhooks if our server responds slowly (timeout). To prevent double-crediting, the idempotency check and status update MUST be in the same database transaction:

**Processing Logic:**
```typescript
// Pseudo-code with Prisma $transaction for atomicity
async processWebhook(payload: WebhookDto) {
  const result = await this.prisma.$transaction(async (tx) => {
    // 1. Check idempotency INSIDE transaction (atomic lock)
    const existing = await tx.invoice.findFirst({
      where: { external_transaction_id: payload.transactionId }
    });
    if (existing) return { status: 'already_processed' };

    // 2. Find invoice by payment_reference
    const paymentRef = payload.description.match(regex)?.[0];
    const invoice = paymentRef 
      ? await tx.invoice.findFirst({ where: { payment_reference: paymentRef } })
      : null;
    
    // 3. Handle unmatched payment (Vietnamese users often mistype transfer descriptions)
    if (!invoice) {
      await tx.unmatched_payments.create({
        data: {
          gateway: payload.gateway,
          transaction_id: payload.transactionId,
          amount: payload.amount,
          sender_name: payload.senderName,
          description: payload.description,
          received_at: new Date(),
          raw_payload: payload,
          status: 'pending',
        }
      });
      return { status: 'unmatched_payment_stored' };
    }

    // 4. Update invoice + store webhook payload (atomic with idempotency check)
    await tx.invoice.update({
      where: { id: invoice.id },
      data: {
        status: 'paid',
        paid_amount: payload.amount,
        paid_at: new Date(),
        external_transaction_id: payload.transactionId,
        raw_gateway_response: payload,
      }
    });

    return { status: 'matched', invoiceId: invoice.id };
  });

  // 5. Emit event + queue notification (OUTSIDE transaction)
  if (result.status === 'matched') {
    this.socketGateway.emit('payment.completed', { invoiceId: result.invoiceId });
    await this.notificationQueue.add('send', { type: 'payment_confirmed', ... });
  } else if (result.status === 'unmatched_payment_stored') {
    // Notify accountants about unmatched payment for manual review
    await this.notificationQueue.add('send', { 
      type: 'unmatched_payment', 
      roles: ['accountant', 'admin'] 
    });
  }
}
```

### 1.4 Unmatched Payments Handling

**Problem**: Vietnamese residents often mistype transfer descriptions (thiбәҝu chб»Ҝ, sai sб»‘ invoice) or reuse old month's content.

**Solution**: `unmatched_payments` table stores transactions that couldn't be auto-matched, allowing Accountants to manually reconcile.

```prisma
model unmatched_payments {
  id             String   @id @default(uuid())
  gateway        String   @db.VarChar(20)  // 'payos' | 'casso' | 'sepay'
  transaction_id String   @unique          // External tx ID (idempotency)
  amount         Decimal  @db.Decimal(15, 2)
  sender_name    String?  @db.VarChar(200)
  description    String?                   // Original transfer description
  received_at    DateTime @db.Timestamptz(6)
  raw_payload    Json                      // Full webhook for audit
  
  // Reconciliation status
  status         String   @default("pending") @db.VarChar(20) // 'pending' | 'matched' | 'rejected'
  matched_invoice_id String? @db.Uuid
  matched_by     String?  @db.Uuid         // Accountant who matched
  matched_at     DateTime? @db.Timestamptz(6)
  rejection_reason String?
  
  created_at     DateTime @default(now()) @db.Timestamptz(6)
  
  @@index([status])
  @@index([received_at])
  @@index([amount])  // Helps accountant filter by amount
}
```

**Accountant Workflow:**
1. Accountant sees list of `unmatched_payments` with status `pending`
2. Searches for matching invoice by amount, date, or sender name
3. Clicks "Match to Invoice" вҶ’ system updates invoice as paid + marks unmatched_payment as `matched`
4. Alternatively, marks as `rejected` with reason (e.g., "Duplicate", "Not our customer")

### 1.5 Manual Re-sync (Reconciliation Job)

**Problem**: If server is down during webhook delivery, transactions may be lost (gateways retry but not forever).

**Solution**: Manual + scheduled reconciliation via Gateway API.

```typescript
// Reconciliation service
async reconcileLast24Hours(gateway: 'payos' | 'casso' | 'sepay') {
  const adapter = this.getGatewayAdapter(gateway);
  const transactions = await adapter.fetchRecentTransactions({ hours: 24 });
  
  for (const tx of transactions) {
    // Check if already processed
    const exists = await this.prisma.invoice.findFirst({
      where: { external_transaction_id: tx.transactionId }
    });
    if (exists) continue;
    
    // Check if in unmatched_payments
    const unmatched = await this.prisma.unmatched_payments.findFirst({
      where: { transaction_id: tx.transactionId }
    });
    if (unmatched) continue;
    
    // Process as if webhook just arrived
    await this.processWebhook(tx);
  }
}
```

**Trigger Options:**
- Manual button in Accountant dashboard: "Re-sync last 24h"
- Scheduled BullMQ job: Daily at 6 AM
- Admin can trigger via API: `POST /api/v1/payments/reconcile`

### 2.2 Zalo ZNS (Zalo Notification Service) Integration

**Zalo ZNS** sends transactional messages via Zalo Official Account. Higher read rates than push notifications for Vietnamese users.

**Requirements:**
- Registered Zalo OA (Official Account) with verified business
- ZNS templates pre-approved by Zalo
- User must follow the OA or have Zalo ID linked to account

**Template Examples:**
```json
// payment_confirmed template
{
  "template_id": "12345",
  "template_data": {
    "customer_name": "Nguyб»…n VДғn A",
    "invoice_number": "INV-2026-04-001",
    "amount": "5,000,000 VND",
    "payment_date": "15/04/2026"
  }
}
```

**Channel Priority Logic:**
```typescript
async sendNotification(userId: string, type: string, data: any) {
  const prefs = await this.getUserPreferences(userId);
  const user = await this.getUser(userId);
  
  const channels: NotificationChannel[] = [];
  
  // Priority 1: Zalo ZNS (highest read rate in Vietnam)
  if (prefs.zalo_enabled && user.zalo_id) {
    channels.push({ type: 'zalo_zns', priority: 1 });
  }
  
  // Priority 2: FCM Push (mobile/web)
  if (prefs.push_enabled) {
    channels.push({ type: 'fcm', priority: 2 });
  }
  
  // Priority 3: Email (fallback)
  if (prefs.email_enabled) {
    channels.push({ type: 'email', priority: 3 });
  }
  
  // Send via all enabled channels (not exclusive)
  for (const channel of channels) {
    await this.notificationQueue.add('send', { userId, type, data, channel });
  }
}
```

## 3. Extended RBAC Design

### 3.1 Building-Scoped Permissions

**Problem**: Security guards work in shifts and should only access data for their assigned building (BбәЈo vб»Ү tГІa A khГҙng Д‘Ж°б»Јc xem danh sГЎch cЖ° dГўn tГІa B).

**Solution**: Add `building_assignments` table to scope permissions by building.

```prisma
model user_building_assignments {
  id          String   @id @default(uuid())
  user_id     String   @db.Uuid
  building_id String   @db.Uuid
  role        UserRole // Role scoped to this building
  assigned_at DateTime @default(now())
  assigned_by String?  @db.Uuid
  
  user      users     @relation(fields: [user_id], references: [id], onDelete: Cascade)
  building  buildings @relation(fields: [building_id], references: [id], onDelete: Cascade)
  
  @@unique([user_id, building_id])
  @@index([building_id])
}
```

**Scope Logic:**
- `admin`: Global access (all buildings)
- `building_manager`: Scoped to assigned buildings
- `security`, `housekeeping`, `technician`: Scoped to assigned buildings
- `accountant`: Can be global or building-scoped (configurable)
- `resident`: Implicitly scoped via apartment вҶ’ building relation

### 3.3 Implementation Approach

**Updated Guard with Building Scope:**
```typescript
@Injectable()
export class BuildingScopedGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest();
    const user = request.user;
    const buildingId = request.params.buildingId || request.body.buildingId;
    
    // Admin bypasses all scoping
    if (user.roles.includes(UserRole.admin)) return true;
    
    // Check if user is assigned to this building
    const hasAssignment = user.buildingAssignments.some(
      a => a.building_id === buildingId
    );
    
    return hasAssignment;
  }
}

// Usage
@Roles(UserRole.security)
@UseGuards(BuildingScopedGuard)
@Get('buildings/:buildingId/residents')
async getResidents(@Param('buildingId') buildingId: string) { ... }
```

### 3.4 Legacy Role-based Approach (for non-scoped endpoints)

**Option A: Role-based (current approach)**
```typescript
@Roles(UserRole.admin, UserRole.accountant)
@Get('invoices')
async getInvoices() { ... }
```

**Option B: Permission-based (more flexible)**
```typescript
@RequirePermission('invoices:read')
@Get('invoices')
async getInvoices() { ... }
```

**Decision**: Keep role-based for simplicity. Building scope enforced via `BuildingScopedGuard` for relevant endpoints.

### 4.3 Account Linking Logic

```typescript
async handleOAuthCallback(profile: OAuthProfile) {
  // 1. Check if OAuth account already linked
  const existingOAuth = await db.oauthAccount.findUnique({
    where: { provider_provider_user_id: { provider: profile.provider, provider_user_id: profile.id } }
  });
  if (existingOAuth) {
    // Login existing user
    return this.authService.generateTokens(existingOAuth.user_id);
  }

  // 2. Check if email exists in users table
  const existingUser = await db.user.findUnique({
    where: { email: profile.email }
  });
  if (existingUser) {
    // Link OAuth to existing user
    await db.oauthAccount.create({
      data: { user_id: existingUser.id, provider: profile.provider, provider_user_id: profile.id }
    });
    return this.authService.generateTokens(existingUser.id);
  }

  // 3. Create new user + OAuth account
  const newUser = await db.user.create({
    data: { email: profile.email, full_name: profile.name, role: UserRole.resident, ... }
  });
  await db.oauthAccount.create({
    data: { user_id: newUser.id, provider: profile.provider, provider_user_id: profile.id }
  });
  return this.authService.generateTokens(newUser.id);
}
```
