# AI Project Constitution - Vully

## 1. Mission and Scope
Vully is a full-stack apartment management platform in a monorepo.
This file is the single source of truth for AI agents working in this repository.

Primary goals for AI work:
- Minimize token usage while preserving correctness.
- Prefer layered context docs before broad code traversal.
- Keep FE-BE contracts aligned and auditable.

## 3. Canonical Data Flow
Use this order when reasoning about features and bugs:
1. Next.js UI and route entry (`apps/web/src/app`)
2. Feature components/hooks (`apps/web/src/components`, `apps/web/src/hooks`)
3. Shared contracts/types (`packages/shared-types`)
4. NestJS controller boundary (`apps/api/src/modules/*/*.controller.ts`)
5. NestJS service orchestration (`apps/api/src/modules/*/*.service.ts`)
6. Persistence and integrations (Prisma/PostgreSQL, Redis/BullMQ, Socket.IO)
7. Side effects (notifications, queue jobs, background processors, audit logs)

## 4. Module Map (Business Flow View)
- Identity: Auth + users + RBAC guard all protected APIs.
- Apartments domain: Buildings, apartments, contracts, parking, policies, access cards, payment schedules.
- Billing domain: Invoices, meter readings, utility pricing, billing jobs, payment sync.
- Incidents domain: Ticket lifecycle, comments, technician workflows, realtime updates.
- Stats domain: Aggregation and dashboard summaries with Redis caching.
- AI assistant domain: RAG and assistant orchestration for policy and operational Q&A.
- Notifications/payments-webhook: external event intake and delivery workflows.

## 5. Coding Standards for AI Edits

### Naming and File Conventions
- Components: PascalCase.
- Hooks: camelCase with `use` prefix.
- Store files: camelCase with `Store` suffix.
- Backend DTOs: class-validator decorated DTO classes.

### Error Handling
- Backend must use standardized exception flow and shared filters.
- API responses must preserve `{ data, meta, errors }` envelope conventions where applicable.
- Do not swallow errors; include actionable context in logs.

### Dependency Injection and Architecture
- Backend services are stateless and injected via NestJS DI.
- Module boundaries must be respected; avoid cross-module tight coupling.
- Heavy/async operations must use BullMQ instead of blocking request handlers.

### Frontend Data Access
- Prefer TanStack Query hooks for API calls.
- Avoid ad-hoc fetch in components where query/mutation hooks exist.
- Keep URL-filtered views synchronized with nuqs where relevant.

## 6. Retrieval and Token-Saving Rules (Mandatory)

### Exclusions by Default
Do not scan these folders for context unless explicitly needed:
- `node_modules`
- `dist`
- `.next`
- `coverage`

### Retrieval Priority (Context-First)
For module-scoped tasks, read in this order:
1. `README.context.md` near relevant feature/module
2. `_module.md` in backend module folder
3. `docs/api-contracts.md` for FE-BE contract tasks
4. `.project-context.md` (this file)
5. Source code files (`.ts`, `.tsx`) only as needed

### Search Strategy
- Start narrow (module-level) then expand.
- Avoid loading full folders when a context doc exists.
- For FE tasks, prioritize colocated context docs under current structure (`app`, `components`, `hooks`).

## 7. OpenSpec and Change Governance
- Use OpenSpec proposal/apply flow for architectural or capability changes.
- Proposal stage must not include runtime code changes.
- Apply stage should keep edits minimal and traceable to tasks.
- Validate changes with strict OpenSpec validation before handoff.

## 8. GitNexus Operational Notes
- If index is stale after major edits, refresh index before relying on retrieval answers.
- Prefer context docs and contracts in retrieval prompts before broad graph/code exploration.
- If GitNexus supports weighting config, prioritize markdown context docs over implementation files.

## 9. Documentation Freshness Rule
Whenever one of these changes occurs, update the relevant context docs in the same work item:
- API request/response contract changes
- New queue/event side effects
- Cross-module dependency changes
- Feature ownership or flow changes

Freshness target:
- Core modules (identity, apartments, billing, incidents, ai-assistant, notifications): update in same PR/change.
- Other modules: update before release cut.
